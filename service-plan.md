# Memory 서비스 기획서

최종 수정일: 2026-05-03

## 1. 서비스 목표

Memory는 웹 기반 개인 일기 서비스다. 핵심 목표는 사용자가 노트북과 모바일 어디서든 빠르고 안정적으로 기록할 수 있게 만드는 것이다.

서비스의 가장 중요한 품질은 다음 네 가지다.

1. 빠르게 열린다.
2. 타이핑이 끊기지 않는다.
3. 기록이 사라지지 않는다.
4. 사용자가 자기 데이터를 쉽게 가져갈 수 있다.

Memory는 기능이 많은 앱보다 오래 믿고 쓸 수 있는 앱을 지향한다. 따라서 서비스 기획의 중심은 "무엇을 더 넣을까"보다 "기록 경험을 방해하는 요소를 어떻게 줄일까"에 있다.

## 2. 서비스 핵심 사용 흐름

### 2.1 데스크톱 기본 흐름

사용자는 노트북에서 긴 일기를 쓴다.

기본 흐름:

1. 웹앱 접속
2. 오늘 화면 자동 진입
3. 중앙 에디터에 바로 타이핑
4. 로컬 자동저장
5. 기분 아이콘, 사용자 태그, 사진을 필요할 때만 추가
6. 저장 상태 확인
7. 과거 기록은 검색이나 캘린더로 이동

중요한 UX:

- 첫 화면에서 클릭 없이 바로 쓸 수 있어야 한다.
- 에디터는 입력 지연이 없어야 한다.
- 자동저장 표시가 있어야 하지만 글쓰기를 방해하면 안 된다.
- 좌우 패널은 보조 역할이고, 중앙 에디터가 항상 우선이다.

### 2.2 모바일 기본 흐름

사용자는 폰에서 빠르게 하루의 흔적을 남긴다.

기본 흐름:

1. 웹앱 또는 홈 화면 PWA 실행
2. 오늘 화면 진입
3. 기분 선택
4. 사용자 태그 선택
5. 한 줄 메모 또는 사진 추가
6. 완료
7. 나중에 노트북에서 장문으로 확장

중요한 UX:

- 10초 안에 저장까지 끝나야 한다.
- 긴 본문은 선택 사항이어야 한다.
- 사진 첨부 후 앱이 멈춘 느낌이 없어야 한다.
- 오프라인이어도 기록 가능해야 한다.

### 2.3 회고 흐름

사용자는 과거 기록을 다시 찾아본다.

기본 흐름:

1. 캘린더에서 날짜 선택
2. 해당 날짜의 기록 확인
3. 검색어 또는 태그로 과거 기록 탐색
4. 월간 회고에서 감정과 태그 패턴 확인
5. 필요하면 PDF나 Markdown으로 내보내기

중요한 UX:

- 검색은 빠르게 반응해야 한다.
- 긴 목록 스크롤이 버벅이면 안 된다.
- 오래된 기록도 접근 가능해야 한다.

## 3. MVP 서비스 범위

### 3.1 MVP에 포함할 기능

MVP는 "로컬 우선 일기 서비스"로 출시한다.

포함 기능:

- 오늘 일기 작성
- 자동저장
- 로컬 저장
- 빠른 감정 기록
- 사용자 태그
- 사진 첨부
- 캘린더
- 검색
- 태그 필터
- 감정 필터
- Markdown 내보내기
- JSON 내보내기
- PDF 또는 인쇄용 보기
- 프라이버시 대시보드
- 반응형 데스크톱/모바일 UI

### 3.2 MVP에서 제외할 기능

초기 버전에서 제외한다.

- 로그인 필수화
- 완전한 클라우드 동기화
- AI 요약
- AI 챗
- 공개 공유
- 커플/가족 공유
- 네이티브 앱
- 위젯
- 지도
- 건강 데이터 연동
- 캘린더 자동 연동
- 고급 꾸미기
- 스티커 마켓

제외 이유:

- 초기에는 기록 신뢰와 성능이 더 중요하다.
- 동기화와 AI는 신뢰 리스크가 크다.
- 공유와 꾸미기는 제품의 방향을 흐릴 수 있다.

## 4. 성능 기획 원칙

웹 기반 서비스이므로 "버벅거림 없음"을 핵심 제품 요구사항으로 둔다.

### 4.1 성능 목표

목표:

- 앱 첫 진입 후 오늘 화면이 빠르게 보인다.
- 타이핑 중 입력 지연이 없다.
- 날짜 이동이 즉각 반응한다.
- 검색 입력 후 결과가 빠르게 갱신된다.
- 사진을 첨부해도 UI가 멈추지 않는다.
- 오래된 기록이 많아져도 목록 스크롤이 부드럽다.

정량 목표:

- 주요 상호작용은 100ms 안에 시각적 피드백을 준다.
- Core Web Vitals의 INP는 좋은 상태를 목표로 한다.
- 긴 작업은 메인 스레드를 50ms 이상 막지 않도록 쪼갠다.
- 모바일 저사양 기기에서도 타이핑과 스크롤이 우선 반응해야 한다.

### 4.2 가장 중요한 성능 기준

일기 앱에서 가장 중요한 성능은 첫 로딩보다 입력 반응성이다.

우선순위:

1. 타이핑 반응성
2. 자동저장 중 UI 안정성
3. 검색 입력 반응성
4. 캘린더 이동 반응성
5. 사진 첨부 중 UI 유지
6. 긴 목록 스크롤 성능
7. 첫 로딩 속도

첫 로딩이 조금 빠른 것보다, 일기를 쓰는 동안 한 번도 멈추지 않는 것이 더 중요하다.

## 5. 성능을 위한 제품 설계

### 5.1 에디터 설계

에디터는 가장 가볍게 만든다.

원칙:

- MVP에서는 무거운 블록 에디터를 피한다.
- Markdown 기반 또는 가벼운 rich text를 우선한다.
- 입력할 때마다 전체 문서를 렌더링하지 않는다.
- 자동저장은 debounce한다.
- 저장 작업은 UI 입력 흐름과 분리한다.
- 글자 수, 태그 추천, AI 분석 같은 부가 계산은 입력 중 실행하지 않는다.

금지:

- 타이핑마다 Markdown 전체 파싱
- 타이핑마다 검색 인덱스 재생성
- 타이핑마다 AI 분석
- 타이핑마다 무거운 통계 계산
- 에디터 내부에 과한 애니메이션

### 5.2 자동저장 설계

자동저장은 신뢰 기능이지만, 성능을 해치면 안 된다.

권장 흐름:

1. 입력 즉시 메모리 상태 업데이트
2. 짧은 지연 후 IndexedDB에 저장
3. 저장 성공 시 상태 표시 업데이트
4. 백업이나 동기화는 별도 큐에서 처리

상태 표시:

- 저장 중
- 방금 저장됨
- 오프라인, 로컬 저장됨
- 저장 실패
- 백업 필요

원칙:

- 저장 실패가 입력을 막으면 안 된다.
- 저장이 느려도 타이핑은 계속 가능해야 한다.
- 사용자가 창을 닫거나 새로고침해도 복구 가능해야 한다.

### 5.3 검색 설계

검색은 기록이 쌓일수록 중요해진다.

MVP 검색:

- 제목
- 본문
- 태그
- 감정
- 날짜

성능 원칙:

- 검색 입력은 debounce한다.
- 긴 본문 전체 검색은 worker 또는 별도 인덱스를 고려한다.
- 결과는 페이지네이션 또는 가상 목록으로 보여준다.
- 검색 중에도 입력창은 즉각 반응해야 한다.

초기에는 단순 검색으로 시작하되, 데이터가 많아지는 시점을 대비해 인덱스 구조를 분리한다.

### 5.4 캘린더와 타임라인 설계

캘린더는 월 단위 요약 데이터만 먼저 가져온다.

원칙:

- 월 이동 시 해당 월의 요약만 읽는다.
- 모든 일기 본문을 한 번에 불러오지 않는다.
- 타임라인은 필요한 개수만 렌더링한다.
- 오래된 기록은 스크롤 시 점진적으로 불러온다.

표시 데이터:

- 날짜
- 기록 여부
- 대표 감정
- 사진 여부
- 글자 수 또는 짧은 미리보기

본문 전체는 사용자가 해당 날짜를 열 때 불러온다.

### 5.5 사진과 첨부파일 설계

사진은 웹앱 성능을 쉽게 무너뜨릴 수 있다.

원칙:

- 원본 이미지를 화면에 바로 쓰지 않는다.
- 썸네일을 만든다.
- 큰 이미지는 지연 로딩한다.
- 첨부 직후 압축 또는 리사이징은 가능하면 비동기로 처리한다.
- 사진 목록은 가상화 또는 지연 렌더링한다.

MVP에서는 사진 기능을 욕심내지 않는다.

MVP 기준:

- 빠른 기록에 사진 1장 또는 제한된 여러 장
- 썸네일 생성
- 원본 보관 여부 명확히 표시
- 내보내기 시 첨부 포함

### 5.6 프라이버시 대시보드 성능

프라이버시 대시보드는 많은 계산을 하지 않는다.

표시할 상태:

- 저장 위치
- 마지막 저장
- 마지막 백업
- 동기화 여부
- 암호화 여부
- AI 여부

원칙:

- 앱 시작 시 모든 데이터를 스캔하지 않는다.
- 필요한 상태값을 별도 메타데이터로 저장한다.
- 대시보드를 열 때만 자세한 검사를 실행한다.

## 6. 데이터 구조 기획

기술 상세는 별도 문서에서 확정하되, 서비스 기획상 필요한 데이터 단위는 다음과 같다.

### 6.1 Entry

일기 하나를 의미한다.

포함 정보:

- id
- date
- title
- body
- mood
- tags
- tags
- attachments
- createdAt
- updatedAt
- deletedAt
- version
- saveStatus

### 6.2 Daily Summary

캘린더와 타임라인을 빠르게 보여주기 위한 요약 데이터다.

포함 정보:

- date
- hasEntry
- mood
- tagCount
- attachmentCount
- wordCount
- preview
- updatedAt

### 6.3 Attachment

사진, 음성, 파일 같은 첨부 데이터다.

포함 정보:

- id
- entryId
- type
- localUrl 또는 storageKey
- thumbnailKey
- fileName
- size
- createdAt

### 6.4 Backup Manifest

내보내기와 복원 신뢰성을 위한 메타데이터다.

포함 정보:

- exportId
- createdAt
- entryCount
- attachmentCount
- format
- checksum
- appVersion

### 6.5 Settings

사용자 설정이다.

포함 정보:

- theme
- writingFont
- autoSaveEnabled
- backupReminder
- aiEnabled
- encryptionEnabled
- syncEnabled

## 7. 화면별 서비스 기획

### 7.1 오늘 화면

목적:

- 사용자가 가장 빠르게 기록을 시작하게 한다.

구성:

- 중앙 에디터
- 날짜 선택
- 저장 상태
- 감정 선택
- 사용자 태그
- 사진 첨부
- 오늘의 짧은 글감

성능 주의:

- 에디터 렌더링을 최우선으로 한다.
- 오른쪽 패널 데이터는 늦게 떠도 된다.
- 글감, 통계, 백업 상태는 지연 로딩 가능하다.

### 7.2 빠른 기록

목적:

- 긴 글 없이도 하루 기록을 완료하게 한다.

구성:

- mood picker
- tag chips
- one-line memo
- photo attach
- done action

성능 주의:

- mood tap 후 즉시 선택 상태가 보여야 한다.
- 사진 처리 중에도 완료 상태를 막지 않는다.
- 첨부 처리는 백그라운드 큐로 넘긴다.

### 7.3 캘린더

목적:

- 날짜별 기록 여부와 감정 흐름을 보여준다.

구성:

- 월간 캘린더
- 날짜별 mood marker
- 선택 날짜 preview
- 월 이동

성능 주의:

- 월별 요약만 로드한다.
- 모든 Entry 본문을 읽지 않는다.

### 7.4 검색

목적:

- 과거 기록을 빠르게 찾게 한다.

구성:

- 검색창
- 필터
- 결과 목록
- 미리보기

성능 주의:

- 입력마다 즉시 화면 피드백을 준다.
- 실제 검색은 debounce한다.
- 결과 목록은 필요한 만큼만 렌더링한다.

### 7.5 보관함

목적:

- 내보내기, 백업, 가져오기, 복원을 관리한다.

구성:

- Markdown export
- JSON export
- PDF export
- 최근 export 기록
- 복원 테스트

성능 주의:

- 큰 export는 즉시 실행하지 말고 진행 상태를 보여준다.
- export 생성은 worker 또는 chunk 처리를 고려한다.
- UI는 export 중에도 반응해야 한다.

### 7.6 프라이버시

목적:

- 사용자가 데이터 상태를 이해하게 한다.

구성:

- 저장 위치
- 마지막 저장
- 마지막 백업
- 동기화 상태
- 암호화 상태
- AI 상태

성능 주의:

- 모든 데이터 검사는 사용자가 요청할 때만 실행한다.
- 기본 화면은 캐시된 메타데이터를 보여준다.

## 8. 초기 서비스 아키텍처 방향

### 8.1 권장 시작 구조

초기 서비스는 계정 없는 로컬 우선 웹앱으로 시작한다.

구조:

- 클라이언트 앱
- IndexedDB 로컬 저장
- Service Worker 기반 오프라인 캐시
- Export 파일 생성
- 나중에 선택형 백업 또는 동기화 추가

이유:

- 계정 없이 바로 써볼 수 있다.
- 초기 개발 복잡도가 낮다.
- 기록 신뢰를 먼저 검증할 수 있다.
- 동기화 실패 리스크를 뒤로 미룰 수 있다.

### 8.2 나중에 추가할 서버 역할

서버는 처음부터 모든 것을 책임지지 않는다.

나중에 필요한 서버 역할:

- 계정
- 암호화된 동기화
- 백업 저장
- 결제
- 사용자 설정 동기화
- 선택형 AI 기능

원칙:

- 서버가 없어도 기본 기록은 가능해야 한다.
- 서버 기능은 편의 기능이어야지 기록의 필수 조건이 아니어야 한다.

## 9. 버벅임 방지를 위한 금지사항

MVP에서 피해야 할 구현/제품 선택:

- 무거운 블록 에디터
- 첫 화면에서 전체 기록 로드

## 12. 운영과 신뢰

일기 서비스는 기능보다 신뢰 운영이 중요하다.

초기부터 필요한 운영 원칙:

- 개인정보 처리 방침을 짧고 명확하게 쓴다.
- 데이터가 어디 저장되는지 설명한다.
- export 방법을 쉽게 제공한다.
- 장애나 저장 오류가 있으면 숨기지 않는다.
- 백업을 권장하되 공포를 조장하지 않는다.
- AI 기능을 도입한다면 별도 동의와 설명을 제공한다.

## 13. 최종 서비스 방향

Memory의 실제 서비스 방향은 다음으로 확정한다.

> 계정 없이 바로 시작할 수 있는 로컬 우선 웹 일기장.  
> 노트북에서는 장문 작성이 끊기지 않고, 모바일에서는 10초 기록이 가능하다.  
> 기록은 자동저장되고, 검색 가능하며, 언제든 Markdown/JSON/PDF로 내보낼 수 있다.  
> 디자인은 따뜻하고 조용한 저널 워크스페이스이며, AI와 클라우드는 나중에 선택 기능으로만 추가한다.
