# Memory 기획 PRD

최종 수정일: 2026-05-03

## 1. 문서 목적

이 문서는 Memory 프로젝트를 처음 보는 사람이나 다른 AI가 제품의 방향을 한 번에 이해할 수 있도록 만든 기획 중심 PRD다.

기술 구현 사양서가 아니라, 다음 질문에 답하는 제품 기준 문서다.

- 무엇을 만들 것인가?
- 누구를 위한 제품인가?
- 왜 웹/PWA로 시작하는가?
- 어떤 기능은 반드시 있어야 하는가?
- 어떤 기능은 있으면 좋지만 나중으로 미뤄야 하는가?
- 어떤 기능은 만들지 않는 것이 좋은가?
- 어떤 디자인 방향을 따라야 하는가?
- 제품이 성공했다는 기준은 무엇인가?

이 문서는 이후 기술 PRD, 와이어프레임, 디자인 시스템, 개발 작업 목록, MVP 구현 계획을 만들 때 기준 문서로 사용한다.

## 2. 제품 한 줄 정의

Memory는 노트북에서는 깊게 쓰고, 모바일에서는 10초 만에 남기며, 모든 기록을 사용자가 소유할 수 있는 웹 기반 개인 일기장이다.

핵심 문장:

> 폰에서는 빠르게 기록하고, 노트북에서는 길게 쓰고, 모든 일기는 내가 소유하는 조용한 기억 저장소.

## 3. 제품 요약

Memory는 안드로이드, 아이폰, 태블릿, 노트북에서 모두 사용할 수 있는 웹 기반 일기 앱이다.

초기 제품은 반응형 웹 앱으로 만들고, 이후 PWA로 확장한다. 사용자는 앱스토어 설치 없이 웹에서 바로 시작할 수 있고, 필요하면 홈 화면에 설치해서 앱처럼 사용할 수 있다.

이 제품은 모바일 전용 일기 앱이 아니다. 오히려 노트북으로 긴 글을 쓰기 편해야 한다. 다만 모바일에서는 하루의 기분, 활동, 사진, 한 줄 메모를 빠르게 남길 수 있어야 한다.

Memory가 지켜야 할 중심 가치는 다음과 같다.

- 쓰기 쉽다.
- 기록이 사라지지 않는다.
- 검색하고 다시 볼 수 있다.
- 내보낼 수 있다.
- 사적인 공간이라는 신뢰가 있다.
- AI, 공유, 꾸미기 같은 부가 기능보다 기록 보존이 우선이다.

## 4. 왜 이 제품을 만드는가

시장조사와 사용자 조사를 통해 일기 앱은 크게 두 부류로 나뉜다는 점을 확인했다.

첫 번째는 Day One, Journey, Diarium 같은 장문 일기 및 라이프로그 앱이다. 이들은 기능이 많고 기록 보관에 강하지만, 사용자는 구독 가격, 동기화 오류, 데이터 유실, 내보내기 제한, 플랫폼 종속에 불만을 느낀다.

두 번째는 Daylio, DailyBean, MOODA, 세줄일기 같은 감정 기록 앱이다. 이들은 빠른 기록, 귀여운 감정 표현, 캘린더, 통계, 꾸미기에서 강하지만, 장문 작성, 웹/데스크톱 사용성, 백업, 검색, 데이터 소유권 측면은 약한 경우가 많다.

Memory는 이 둘 사이의 빈틈을 노린다.

- 감정기록 앱처럼 쉽게 시작할 수 있어야 한다.
- 전문 일기 앱처럼 오래 보관할 수 있어야 한다.
- 웹앱답게 노트북 입력 경험이 좋아야 한다.
- 사용자가 자기 기록을 언제든 꺼낼 수 있어야 한다.

## 5. 제품 포지셔닝

Memory는 로컬 우선, 글쓰기 우선의 개인 기억 저장소다.

Memory는 다음에 가깝다.

- 개인 일기장
- 웹/PWA 다이어리
- 프라이빗 메모리 아카이브
- 감정 기록이 가능한 장문 저널

Memory는 다음이 아니다.

- SNS
- 공개 일기 커뮤니티
- AI 상담사
- Notion 대체재
- 업무 생산성 도구
- 치료나 진단 목적의 정신건강 앱
- 콘텐츠 발행 플랫폼

## 6. 핵심 제품 약속

Memory가 사용자에게 약속해야 하는 것은 기능의 양이 아니라 신뢰다.

1. 쓴 글은 사라지지 않는다.
2. 사용자는 자기 기록을 소유한다.
3. 노트북에서 쓰기 편하다.
4. 모바일에서는 빠르게 남길 수 있다.
5. 검색과 회고가 쉽다.
6. 백업과 내보내기가 당연히 가능하다.
7. AI는 기본값이 아니라 선택 사항이다.
8. 제품은 조용하고 사적인 분위기를 유지한다.

## 7. 타깃 사용자

### 7.1 장문 기록 사용자

노트북으로 글을 쓰는 것을 선호하는 사용자다. 하루를 돌아보며 문장과 문단으로 기록하고 싶어 한다.

필요한 것:

- 큰 글쓰기 화면
- 키보드 중심 UX
- 자동저장
- 날짜 이동
- 검색
- 태그
- 백업
- 내보내기

불편해하는 것:

- 모바일에서 긴 글 쓰기
- 복잡한 꾸미기 중심 UI
- 저장 여부가 불명확한 앱
- 구독하지 않으면 기록을 제대로 꺼내지 못하는 구조
- AI가 내 글을 읽는다는 불안

### 7.2 빠른 감정 기록 사용자

매일 긴 일기를 쓰지는 않지만, 기분 아이콘이나 사용자 태그를 간단히 남기고 싶어 하는 사용자다.

필요한 것:

- 기분 선택
- 사용자 태그 선택
- 한 줄 메모
- 사진
- 알림
- 월간 감정 흐름
- 부담 없는 기록 방식

불편해하는 것:

- 빈 화면 앞에서 무엇을 써야 할지 모르는 부담
- 저장 중 광고
- 너무 긴 입력 절차
- 하루를 놓쳤을 때 죄책감을 주는 streak

### 7.3 프라이버시와 장기 보관을 중시하는 사용자

몇 년치 기록을 안전하게 보관하고 싶은 사용자다. 앱이 사라지거나 가격 정책이 바뀌어도 자신의 기록을 가져갈 수 있기를 원한다.

필요한 것:

- 로컬 우선 저장
- 표준 포맷 내보내기
- 백업 상태 표시
- 암호화
- AI 끄기
- 데이터 삭제와 이동 권한

불편해하는 것:

- 클라우드 전용 구조
- 불투명한 동기화
- 내보내기 제한
- 독점 포맷
- AI 기능 강제

## 8. 사용자 핵심 과업

사용자는 Memory를 통해 다음 일을 해결하고 싶어 한다.

### 기능적 과업

- 하루를 마무리하며 긴 글을 쓴다.
- 낮에 떠오른 감정이나 사건을 빠르게 기록한다.
- 과거의 특정 기억을 검색한다.
- 월별로 감정과 사용자 태그 흐름을 돌아본다.
- 기기를 바꾸거나 서비스를 떠날 때 데이터를 내보낸다.
- 인터넷이 없어도 기록한다.

### 감정적 과업

- 내 일기가 안전하다고 느낀다.
- 아무에게도 평가받지 않고 쓴다.
- 매일 쓰지 못해도 부담을 느끼지 않는다.
- 나중에 돌아볼 수 있다는 안심을 느낀다.
- 내 기록이 내 것이라는 통제감을 가진다.

### 사회적 과업

기본 사용은 완전히 개인적이다. 다만 나중에는 연인, 가족, 가까운 친구, 학급처럼 좁은 관계 안에서 일부 기록을 공유하고 싶어 할 수 있다.

넓은 공개 공유는 핵심 방향이 아니다.

## 9. 제품 원칙

### 9.1 기록 보존이 최우선이다

일기 앱에서 가장 치명적인 실패는 기능 부족이 아니라 기록 유실이다. 자동저장, 복구, 백업, 내보내기는 부가 기능이 아니라 제품의 기반이다.

### 9.2 첫 화면은 바로 쓸 수 있어야 한다

사용자가 앱을 열면 랜딩 페이지나 홍보 화면이 아니라 오늘의 기록 화면을 만나야 한다.

### 9.3 모바일과 데스크톱의 역할을 분리한다

데스크톱은 장문 작성과 검색, 정리에 최적화한다. 모바일은 빠른 캡처와 간단한 회고에 최적화한다.

### 9.4 데이터 소유권은 무료 핵심 기능이다

기본 내보내기 기능은 유료 장벽 뒤에 두면 안 된다. 사용자는 자신의 글을 언제든 가져갈 수 있어야 한다.

### 9.5 프라이버시는 보이는 기능이어야 한다

단순히 "안전합니다"라고 말하는 것이 아니라, 데이터가 어디 저장되는지, 마지막 백업이 언제인지, AI가 켜져 있는지, 암호화가 적용되는지 보여줘야 한다.

### 9.6 AI는 선택 기능이다

AI 요약, 감정 분석, 회고 질문은 나중에 유용할 수 있다. 그러나 일기 앱에서 AI는 민감하다. 기본값은 꺼짐이어야 하고, 사용자가 명시적으로 켤 때만 동작해야 한다.

### 9.7 습관 기능은 부드러워야 한다

연속 기록을 강요하거나 하루 빠졌다고 실패감을 주면 안 된다. "이번 달 12일 기록"처럼 유연한 피드백이 더 적합하다.

### 9.8 제품은 조용해야 한다

Memory는 사용자의 주의를 빼앗는 앱이 아니라 기록을 도와주는 앱이다. 과한 애니메이션, 장식, 게임화, 알림을 피한다.

## 10. 전체 범위

### 10.1 우리가 만드는 것

- 웹 기반 개인 일기 앱
- PWA로 확장 가능한 반응형 앱
- 데스크톱 장문 작성 화면
- 모바일 빠른 기록 화면
- 감정 아이콘과 사용자 태그
- 캘린더와 타임라인
- 검색과 필터
- 로컬 우선 자동저장
- 백업과 내보내기
- 프라이버시 상태 대시보드
- 조용하고 신뢰감 있는 디자인 시스템

### 10.2 처음에는 만들지 않는 것

- 네이티브 Android 앱
- 네이티브 iOS 앱
- 공개 SNS 피드
- AI 상담사
- 실시간 협업
- 복잡한 Notion식 데이터베이스
- 템플릿 마켓플레이스
- 건강, 캘린더, SNS 자동 수집 허브
- 위치 기반 라이프로그 전체 자동화

## 11. 기능 우선순위

## 11.1 P0: 반드시 필요한 MVP 기능

### 11.1.1 오늘 쓰기 화면

제품의 중심 화면이다. 사용자는 앱을 열자마자 오늘의 일기를 쓸 수 있어야 한다.

필수 요소:

- 큰 에디터
- 자동저장
- 저장 상태 표시
- 날짜 변경
- 제목 선택 입력
- 본문 작성
- 간단한 서식
- 키보드 단축키
- 모바일에서도 읽고 수정 가능

좋은 경험:

- 사용자가 "문서 작성 앱"처럼 편하게 쓸 수 있어야 한다.
- 글쓰기 영역 주변은 조용해야 한다.
- 감정, 태그, 사진은 보조 영역에 있어야 한다.

### 11.1.2 빠른 기록

사용자가 긴 글을 쓰지 않아도 하루를 남길 수 있어야 한다.

필수 요소:

- 기분 선택
- 사용자 태그 선택
- 한 줄 메모
- 사진 1장 이상
- 오늘 기록과 연결
- 모바일에서 10초 안에 완료 가능한 흐름

좋은 경험:

- Daylio나 DailyBean처럼 부담이 낮아야 한다.
- 나중에 장문 일기로 확장할 수 있어야 한다.

### 11.1.3 로컬 우선 자동저장

네트워크 상태와 관계없이 먼저 로컬에 저장되어야 한다.

필수 요소:

- 입력 중 자동저장
- 새로고침 후 복구
- 브라우저 재시작 후 복구
- 저장 실패 표시
- 마지막 저장 시간 표시

좋은 경험:

- 사용자는 "저장됐나?"를 걱정하지 않아야 한다.
- 상태는 보이되 방해하지 않아야 한다.

### 11.1.4 캘린더와 타임라인

기록이 쌓이면 날짜별로 돌아보는 기능이 필요하다.

필수 요소:

- 월간 캘린더
- 기록한 날 표시
- 감정 표시
- 최근 기록 타임라인
- 오늘로 이동
- 과거 날짜에 기록 추가

### 11.1.5 검색과 태그

오래 쓸수록 검색은 핵심 기능이 된다.

필수 요소:

- 본문 검색
- 날짜 필터
- 태그 필터
- 감정 필터
- 검색 결과 미리보기

### 11.1.6 백업과 내보내기

Memory의 신뢰를 만드는 기능이다.

필수 요소:

- JSON 내보내기
- Markdown 내보내기
- PDF 또는 인쇄용 보기
- 사진과 첨부 포함 여부 표시
- 날짜, 감정, 태그 같은 메타데이터 포함

중요한 원칙:

- 기본 내보내기는 무료 핵심 기능이어야 한다.
- 사용자가 자기 기록을 가져가는 일을 어렵게 만들면 안 된다.

### 11.1.7 프라이버시 대시보드

사용자는 자기 일기가 어떻게 보관되는지 이해해야 한다.

필수 요소:

- 저장 위치
- 마지막 로컬 저장 시간
- 마지막 백업 또는 내보내기 시간
- 동기화 상태
- 암호화 상태
- AI 사용 상태
- 데이터 삭제와 내보내기 진입점

### 11.1.8 반응형 PWA 기반

하나의 웹앱으로 데스크톱과 모바일을 모두 지원해야 한다.

필수 요소:

- 데스크톱 3패널 레이아웃
- 모바일 하단 탭 또는 간단한 주요 내비게이션
- 오프라인 작성
- 홈 화면 설치 가능성을 고려한 구조

## 11.2 P1: MVP 이후 강하게 필요한 기능

### 11.2.1 선택형 동기화 또는 클라우드 백업

동기화는 중요하지만 초기부터 무리하게 완벽할 필요는 없다. 다만 나중에는 여러 기기에서 이어 쓰는 흐름이 필요하다.

필요 요소:

- 사용자가 직접 켜는 방식
- 마지막 동기화 시간
- 동기화 실패 표시

## 12. 없어야 하는 기능

### 12.1 공개 소셜 피드

공개 피드는 제품의 성격을 바꾼다. 개인 기록이 보여주기식 콘텐츠가 될 수 있고, moderation, 신고, 비교, 공개 범위 문제까지 생긴다.

초기 제품에는 넣지 않는다.

### 12.2 기본으로 켜진 AI

일기 앱에서 AI는 민감하다. 사용자가 직접 켜지 않은 AI가 일기를 읽는다고 느끼면 신뢰가 깨진다.

AI는 기본 꺼짐이어야 한다.

### 12.3 저장 흐름의 광고

일기를 저장하는 순간에 광고를 보여주면 사용자 경험이 크게 손상된다. 특히 저장 실패나 데이터 유실 불안과 결합하면 강한 반감을 만든다.

### 12.4 내보내기 유료화

자기 기록을 가져가는 기능은 제품 신뢰의 핵심이다. 기본 export를 유료화하면 사용자에게 lock-in으로 보인다.

### 12.5 클라우드 전용 구조

사용자는 계정을 만들기 전에도 기록할 수 있어야 한다. 처음부터 클라우드 전용이면 로컬 우선이라는 제품 약속이 약해진다.

### 12.6 벌점형 streak

하루 빠졌다고 기록이 끊기거나 실패처럼 보이면 일기 습관에 부담을 준다. 유연한 목표가 더 적합하다.

### 12.7 과도한 데이터베이스 UI

Memory는 Notion이 아니다. 사용자가 구조를 설계해야만 쓸 수 있는 제품이 되어서는 안 된다.

### 12.8 홍보성 첫 화면

앱의 첫 화면은 제품 설명이 아니라 오늘의 기록이어야 한다.

## 16. 디자인 방향

### 16.1 디자인 컨셉

Memory의 디자인은 "조용한 개인 기록실"이어야 한다.

느껴져야 하는 것:

- 보라색 AI 앱 느낌

- 카드가 중첩된 복잡한 UI
- 장식용 랜딩 페이지

### 16.1.1 확정 디자인 레퍼런스

디자인은 임의의 취향으로 만들지 않고, Pinterest에서 찾은 레퍼런스를 기준으로 한다.

메인 레퍼런스:

- [Journaling App Light & Dark Mode](https://www.pinterest.com/pin/journaling-app-light-dark-mode--133419207699656575/)
- 원본 참고: [Elaine Wang - Journaling App Light & Dark Mode](https://dribbble.com/shots/9777881-Journaling-App-Light-Dark-Mode)

이 레퍼런스를 메인으로 삼는 이유:

- 일기 앱에 맞는 차분한 무드가 있다.
- 밝은 모드와 어두운 모드가 모두 어울린다.
- 과하게 귀엽거나 SaaS처럼 차갑지 않다.
- 본문을 읽고 쓰는 경험을 방해하지 않는 색감이다.
- Memory가 지향하는 "조용한 개인 기록실"에 가장 가깝다.

이 레퍼런스에서 가져올 것:

- 따뜻한 오프화이트 배경
- 잉크에 가까운 어두운 본문색
- 낮은 채도의 베이지/브라운 계열 보조색
- 조용한 light/dark mode 전환 감각
- 기록을 읽는 데 집중되는 여백

그대로 따라 하지 않을 것:

- 모바일 화면 구성 전체
- 특정 아이콘 스타일
- 특정 컬러 팔레트의 완전 복제
- 장식적 이미지나 일러스트

보조 레이아웃 레퍼런스:

- [Personal Dashboard](https://www.pinterest.com/pin/personal-dashboard--108367934774537266/)

이 레퍼런스는 데스크톱의 3패널 구조, 정보 배치, 상태 위젯 감각을 참고한다. 다만 Memory는 생산성 대시보드가 아니므로 밀도는 낮추고 중앙 글쓰기 영역을 훨씬 더 크게 둔다.

보조 모바일 감정 기록 레퍼런스:

- [Desert Mood Diary: UI Design Mobile App](https://www.pinterest.com/pin/91057223712956635/)

이 레퍼런스는 모바일 빠른 감정 기록의 감성적 진입감을 참고한다. 다만 Memory 전체를 사막색/베이지색 테마로 만들지는 않는다.

확정 디자인 방향:

> Warm Quiet Journal Workspace

즉, Memory는 따뜻한 오프화이트 기반의 조용한 저널 앱으로 간다. 데스크톱은 글쓰기 중심의 워크스페이스, 모바일은 빠른 감정 기록 중심의 캡처 UI를 가진다.

### 16.2 색상

권장 방향:

- 배경: 밝은 회백색
- 본문: 잉크 블랙 또는 진한 차콜
- 주요 강조: 차분한 녹색
- 보조 강조: 맑은 파랑
- 경고: 따뜻한 앰버
- 위험: 차분한 레드
- 감정 색상: 노랑, 코랄, 블루, 그레이, 그린을 제한적으로 사용

피해야 할 것:

- 보라색 중심 그라디언트
- 베이지와 크림색만으로 이루어진 다이어리 테마
- 짙은 남색/슬레이트 일색의 대시보드
- 갈색/오렌지 nostalgia 테마에 과도하게 기대는 것
- 장식용 구체, 흐린 배경 덩어리, 의미 없는 그래픽

### 16.3 타이포그래피

한국어 일기 앱에서는 글을 읽고 쓰는 감각이 중요하다.

추천 방향:

- UI 폰트: SUIT, IBM Plex Sans KR 계열
- 본문 폰트: Gowun Batang 같은 읽기 좋은 명조 옵션
- 코드/Markdown/상태 정보: monospace는 보조적으로만 사용

원칙:

- 에디터 본문은 오래 읽어도 피로하지 않아야 한다.
- 패널 안의 제목은 작고 명확해야 한다.
- hero급 큰 글자는 앱 내부에서 남발하지 않는다.
- viewport 크기에 따라 폰트가 과하게 변하지 않아야 한다.

### 16.4 레이아웃

데스크톱:

- 3패널 구조를 기본으로 고려한다.
- 중앙 에디터가 가장 중요하다.
- 좌우 패널은 보조 기능이다.

모바일:

- 오늘 기록이 첫 화면이다.
- 빠른 기록은 가장 짧은 경로로 제공한다.
- 캘린더와 검색은 하단 탭으로 접근한다.
- 에디터와 메타데이터 입력은 분리하거나 접을 수 있어야 한다.

### 16.5 컴포넌트 방향

shadcn/ui 기반 구현을 가정한다.

사용하면 좋은 컴포넌트:

- Sidebar
- Resizable
- Calendar
- Command
- Sheet 또는 Drawer
- Dialog
- Tabs
- Badge
- ToggleGroup
- Tooltip
- Separator
- Sonner toast

규칙:

- 기존 shadcn 컴포넌트를 먼저 사용한다.
- 색상은 semantic token을 사용한다.
- raw color class를 남발하지 않는다.
- 카드 안에 카드를 넣지 않는다.
- 카드 radius는 8px 이하로 유지한다.
- 아이콘 버튼에는 tooltip을 제공한다.
- 일반적인 동작은 텍스트 버튼보다 익숙한 아이콘을 우선한다.
- 앱 내부에 기능 설명 문구를 과하게 넣지 않는다.

## 17. 글쓰기 경험

- 자동저장이 된다.
- 장문 작성이 편하다.
- 주변 UI가 방해하지 않는다.
- 날짜를 바꿀 수 있다.
- Markdown 또는 간단한 서식을 지원한다.
- 모바일에서도 읽고 수정할 수 있다.

초기 서식:

- 문단
- 제목
- 굵게
- 기울임
- 리스트
- 인용
- 링크

MVP에서 피할 것:

- 복잡한 블록 에디터

- 과한 툴바

## 18. 빠른 기록 경험

빠른 기록은 에너지가 낮은 날에도 쓸 수 있어야 한다.

이상적인 흐름:

1. 앱을 연다.
2. 기분을 고른다.
3. 사용자 태그를 고른다.
4. 한 줄을 남기거나 생략한다.
5. 사진을 추가하거나 생략한다.
6. 완료한다.

긴 글을 쓰지 않아도 하루가 기록되어야 한다.

피해야 할 것:

- 제목 필수
- 본문 필수
- 너무 많은 단계
- 매번 나오는 글감 질문
- 저장 전에 광고

## 19. 데이터 소유권과 프라이버시

### 19.1 데이터 소유권

사용자는 자기 일기를 언제든 가져갈 수 있어야 한다.

필수 내보내기:

- JSON
- Markdown
- PDF 또는 인쇄용 보기

내보내기에 포함되어야 하는 것:

- 날짜
- 본문
- 제목
- 감정
- 사용자 태그
- 사진과 첨부파일
- 생성일과 수정일

권장 구조:

- 날짜별 Markdown 파일
- 전체 복원용 JSON
- attachments 폴더
- 사람이 읽기 쉬운 PDF

### 19.2 프라이버시

프라이버시는 숨은 설정이 아니라 보이는 화면이어야 한다.

사용자에게 보여줄 것:

- 이 브라우저에 저장됨
- 마지막 저장 시간
- 마지막 백업 시간
- 동기화 켜짐/꺼짐
- 암호화 켜짐/꺼짐
- AI 켜짐/꺼짐
- 데이터 삭제
- 전체 내보내기

### 19.3 AI 원칙

AI는 다음 조건을 만족할 때만 도입한다.

- 기본 꺼짐
- 사용자가 직접 켬
- 어떤 데이터가 사용되는지 설명
- 학습 사용 여부 설명
- AI 결과 삭제 가능
- 가능하면 로컬 또는 프라이버시 보호 방식 우선

AI가 할 수 있는 일:

- 월간 요약
- 감정 패턴 정리
- 글감 제안
- 의미 기반 검색
- "내가 예전에 뭐라고 썼지?" 질의

AI가 하면 안 되는 일:

- 사용자의 동의 없이 일기 읽기
- 기본적으로 모든 기록 분석하기
- 상담사처럼 단정하기
- 건강 상태를 진단하기

## 20. 백업, 동기화, 신뢰성

신뢰성은 기술 기능이 아니라 사용자 경험이다.

보여줘야 하는 상태:

- 로컬 저장됨
- 저장 중
- 오프라인, 로컬에 저장됨
- 백업 필요
- 마지막 백업
- 동기화 연결됨
- 동기화 오류
- 충돌 발생

충돌 처리 원칙:

- 조용히 덮어쓰지 않는다.
- 불확실하면 두 버전을 모두 보존한다.
- 사용자가 나중에 비교하고 정리할 수 있게 한다.

삭제 처리 원칙:

- 즉시 영구 삭제를 피한다.
- 휴지통 또는 되돌리기를 제공한다.
- 전체 삭제 전에는 명확히 경고한다.

## 21. 수익화 방향

수익화는 신뢰를 해치지 않아야 한다.

무료 핵심으로 유지해야 할 것:

- 일기 작성
- 로컬 저장
- 기본 검색
- 캘린더
- 빠른 기록
- 수동 백업
- 기본 내보내기
- 데이터 삭제

유료화 가능성이 있는 것:

- 암호화된 관리형 동기화
- 대용량 미디어 저장
- 자동 백업
- 고급 가져오기
- 고급 회고 리포트
- 프리미엄 테마
- 프라이버시 보호 AI 기능

피해야 할 수익화:

- 내보내기 유료화
- 저장 흐름 광고
- 오래된 기록 잠금
- AI 강제 업셀
- 기본 백업 제한으로 불안 유발

## 22. 성공 기준

### 22.1 제품 성공 기준

초기 제품은 다음을 만족하면 성공적인 방향이다.

- 사용자가 노트북에서 긴 일기를 편하게 쓴다.
- 모바일에서 10초 안에 빠른 기록을 남긴다.
- 새로고침이나 앱 종료 후에도 글이 사라지지 않는다.
- 사용자가 자기 데이터가 어디 저장되는지 이해한다.
- 사용자가 전체 기록을 내보낼 수 있다.
- 과거 기록을 검색으로 찾을 수 있다.
- UI가 조용하고 신뢰감 있게 느껴진다.

### 22.2 MVP 완료 기준

MVP는 다음 기능이 동작할 때 완료로 본다.

- 오늘 쓰기
- 빠른 기록
- 로컬 자동저장
- 저장 상태 표시
- 캘린더
- 검색
- 태그와 감정 필터
- 수동 내보내기
- 프라이버시 대시보드
- 모바일/데스크톱 반응형 UI

반복적으로 확인된 인사이트:

- AI는 선호와 거부가 강하게 갈린다.
- 한국/아시아권 감정 일기 사용자는 귀여운 감정 표현, 사진, 꾸미기, 캘린더에 반응한다.
- 저장 중 광고와 데이터 잠금형 과금은 강한 반감을 만든다.

## 25. 추천 초기 결정

초기 제품은 다음 범위로 시작하는 것이 좋다.

시작할 것:

- 계정 없는 로컬 우선 웹앱
- 데스크톱 장문 작성
- 모바일 빠른 기록
- 감정 아이콘과 사용자 태그
- 캘린더
- 검색
- Markdown, JSON, PDF 내보내기
- 프라이버시 대시보드
- PWA를 고려한 반응형 구조

시작하지 않을 것:

- 완전한 클라우드 동기화
- AI
- 네이티브 앱
- 공개 공유
- 과한 꾸미기
- 복잡한 외부 연동

가장 먼저 검증해야 할 가설:

> 사용자는 "예쁜 일기 앱"보다 "쓰기 편하고, 잃어버리지 않고, 언제든 가져갈 수 있는 일기 앱"을 더 오래 신뢰하고 사용할 것이다.
